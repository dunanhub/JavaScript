## Task 4 — Documentation (Provider Pattern)

В этой лабораторной работе я реализовал шаблон Provider, используя React Context API. Основная проблема, которую я решил, - это детализация prop: когда многим вложенным компонентам требуются одни и те же данные, данные передаются на каждом уровне в качестве prop, даже если промежуточные компоненты их не используют. В примере из учебника (глава 1, “Пример повседневного использования шаблонов проектирования”) шаблон поставщика показан как простое решение для совместного использования общего состояния в глубоком дереве компонентов.

Моя реализация создает "UserContext" с помощью "createContext()" и пользовательского перехватчика "useUser()", который обертывает "useContext(UserContext)". Этот хук включает в себя обработку ошибок: если он вызывается за пределами провайдера, он выдает ошибку. Затем я создал "UserProvider", который хранит фиктивные данные пользователя (имя, адрес электронной почты, роль), разрешения (CanEdit, CanDelete, CanView) и настройки темы (темный режим, размер шрифта). Поставщик передает полное значение контекста дочерним компонентам, чтобы любой компонент мог считывать одно и то же общее состояние.

Чтобы продемонстрировать, что детализация по реквизитам исключена, я использовал три пользовательских компонента: `Dashboard`, `Header` и `UserMenu`. Каждый компонент считывает необходимые данные непосредственно из контекста с помощью `useUser()`, не получая реквизиты от родительских компонентов. Такой подход упрощает обслуживание кода, сокращает повторную передачу prop и упрощает масштабирование в будущем, когда дерево компонентов станет более глубоким.
